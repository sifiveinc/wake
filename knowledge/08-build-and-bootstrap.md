# Build & Bootstrap

Wake is **self-hosted**: a small C++ Makefile builds the wake binary, then wake builds itself (and everything else) from `build.wake`.

## Phase 1: bootstrap (`Makefile`)

`make` (top-level) compiles all C/C++ in `src/` and `tools/` and links:

- `bin/wake` — main binary
- `lib/wake/wakebox`
- `lib/wake/fuse-waked`
- `lib/wake/shim-wake`
- `lib/wake/wake-hash`
- (and others as needed)

External system dependencies (must be installed):
- sqlite3 ≥ 3.6
- libgmp ≥ 4.3
- libfuse ≥ 2.8
- libre2 ≥ 2013
- libncurses ≥ 5.7
- dash ≥ 0.5

Vendored deps (in `vendor/`) include BLAKE2, SipHash, gopt, lemon, whereami, utf8proc.

## Phase 2: self-hosted build (`build.wake`)

Once `bin/wake` exists, `make` delegates to it:

```
make             # ≈ ./bin/wake build default
```

`build.wake` defines build *variants* (sometimes called targets in user docs). Defaults visible there include:

- `default` — native release.
- `static` — fully statically linked binary.
- `debug` — native with debug info, asserts, no optimization.
- `wasm` — WebAssembly (via emscripten).

Run a specific variant:
```bash
./bin/wake build default
./bin/wake build static
./bin/wake build debug
./bin/wake build wasm
```

`src/src.wake` defines compiler/runtime build rules (the big one). `tools/tools.wake` builds tooling. `vendor/vendor.wake` (if present) handles vendored deps.

## Phase 3: install

```bash
make install DESTDIR=$HOME/wake
# or directly:
./bin/wake install $HOME/wake
```

Installs:
- `bin/wake`, `bin/wake-format`, etc.
- `lib/wake/<helper binaries>`
- `share/wake/lib/` (the standard library)
- `share/doc/wake/` (user docs)

## Other targets

| Target | Effect |
|---|---|
| `make tarball` | Build a versioned source tarball. Substitutes the version into `wake.spec.in`, `debian/changelog.in`. Generates `.wakemanifest`. |
| `make vscode` | Build the VSCode extension package (`extensions/vscode/`). |
| `make wasm` | Compile the WebAssembly variant (requires emscripten). |
| `make static` | Static-linked binary. |
| `make format` | Run `clang-format` over C++ and `wake-format` over `.wake` files. |
| `make formatAll` | Same, across the entire repo. |
| `make clean` | Remove build artifacts (preserves `wake.db`). |
| `make test` | Integration tests. |
| `make unittest` | C++ unit tests via `tools/wake-unit`. |
| `make remoteCacheTests` | RSC end-to-end (PostgreSQL required). |

## Configuration files

- `.wakeroot` — workspace marker. Recognized JSON5 keys:
  - `version` — SemVer constraint.
  - `user_config` — path to per-repo user-config JSON.
  - other config keys mirroring `wake.json`.
- `~/.config/wake.json` (or `$XDG_CONFIG_HOME/wake.json`) — global user config:
  - `log_header`, `log_header_source_width`, `log_header_align` — logging format.
  - `max_cache_size`, `low_cache_size` — cache-collection thresholds.
  - `cache_miss_on_failure` — soft fail on cache server errors.
- `.wakemanifest` — list of `.wake` source files included in tarball releases (auto-generated by `make tarball`).
- `.wakeignore` — glob patterns to exclude `.wake` files from compilation (see `share/doc/wake/tutorial.md`).

## CI

`.github/workflows/build.yaml` runs on PRs and pushes:

1. Format checks (`clang-format`, `wake-format`) — see `scripts/`.
2. `make tarball`.
3. `make test`.
4. `make unittest`.
5. `make remoteCacheTests` against a PostgreSQL service container.
6. Documentation generation.
7. Upload artifacts (tarballs, binaries, docs).

Tagged commits (`v*`) additionally produce release artifacts.

## Release procedure

From `share/doc/wake/how-to/create-a-release.adoc`:

1. Bump version in `debian/changelog.in`.
2. PR + merge to master.
3. Tag (`git tag v49.x.x && git push --tags`).
4. CI builds release artifacts; a draft GitHub release appears.
5. Build & upload the VSCode extension to the marketplace separately.
6. Polish release notes; publish.

## Common gotchas

- `make` after a deep change to `src/` may not pick up regenerated parser. If lexer/parser updates aren't reflected, `make clean && make`.
- Bootstrap can break if the host compiler is too old; the project targets C++17.
- FUSE-related tests fail without proper kernel support (Linux); macOS uses different sandbox paths.
- `make test` runs many subprocesses; expect minutes, not seconds.
